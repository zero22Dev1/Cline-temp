# AI Learning Curator

作業から再利用可能な知識だけを抽出し、適切な保存先へ昇格させる。会話履歴や一時的な情報を無制限に保存しない。

## 学習対象

次から学習候補を抽出する。

- ユーザーから受けた訂正、明示的な好み、承認された判断
- 実装、レビュー、テスト、品質ゲートの結果
- 繰り返し発生した失敗と、その原因・有効だった修正
- プロジェクト固有の命名、構成、コマンド、設計パターン
- 仕様とコードの差異、および確定した扱い
- 作業を短縮または安全にした手順

単発の偶然、未検証の推測、秘密情報、個人情報、巨大なログは学習対象にしない。

## 学習候補の評価

各候補に次を付ける。

| 項目 | 内容 |
|---|---|
| Learning | 再利用可能な知識 |
| Evidence | ユーザー判断、ファイル、テスト、レビューなどの根拠 |
| Confidence | `High / Medium / Low` |
| Scope | `Task / Feature / Project / Team` |
| Reuse | 再利用が見込まれる場面 |
| Risk | 誤って一般化した場合の影響 |
| Destination | 推奨保存先 |
| Action | `Promote / Keep Candidate / Reject / Replace` |

次をすべて満たす候補を優先して昇格する。

- 根拠を示せる
- 将来の複数タスクで再利用できる
- 適用範囲を限定できる
- 既存ルールや仕様と矛盾しない
- 保存によるノイズより再利用価値が高い

## 保存先を選ぶ

- 現在の焦点、次の作業、短中期の前提: `memory-bank/activeContext.md`
- 完了事項、未完了事項、検証状態: `memory-bank/progress.md`
- 技術構成、コマンド、開発規約: `memory-bank/techContext.md`
- 安定した目的とプロジェクト範囲: `memory-bank/projectbrief.md`
- 用語と定義: `docs/ai/glossary.md`
- 設計判断と却下案: `docs/ai/adr/`
- 未確定事項: `docs/ai/unknowns/`
- 学習候補と評価記録: `docs/ai/learnings/YYYYMMDD-<topic>.md`
- 常時適用すべき短いルール: `.clinerules/`
- 繰り返し実行する専門手順: `.cline/skills/`

同じ知識を複数箇所へそのまま複製しない。参照で十分ならリンクする。

## 実行手順

1. 対象作業の依頼、差分、レビュー、検証結果、ユーザー訂正を読む。
2. 既存の `memory-bank/`、`docs/ai/`、`.clinerules/`、関連Skillを確認する。
3. 新規学習、既存知識の強化、矛盾、陳腐化を分類する。
4. 学習候補を評価表へ記録する。
5. `High`かつ低リスクの候補だけを適切な保存先へ反映する。
6. `Medium`は候補として残し、追加の再現またはユーザー確認を待つ。
7. `Low`、根拠なし、単発情報はRejectする。
8. 更新後に重複、矛盾、古い記述、秘密情報がないか確認する。
9. 変更ファイルと昇格・保留・却下理由を報告する。

## 昇格ルール

### `.clinerules/`への昇格

次をすべて満たす場合だけ行う。

- 原則としてすべての作業で有効
- 短く明確に記述できる
- 例外が少ない
- 既存ルールと競合しない

特定作業だけの長い手順はSkillへ配置する。

### Skillへの昇格

次に該当する場合は `/cline-skill-builder` を使って、新規作成または既存Skill更新を行う。

- 同じ手順を複数回使用した
- 入力、処理、出力、停止条件が定義できる
- 専門的なチェックリストやツール手順がある
- 常時ルールとして読み込むと重い

一度成功しただけの手順を新規Skillへ昇格しない。

### ADRへの昇格

採用案、却下案、理由、影響、見直し条件を説明できる設計判断だけを記録する。

## 矛盾と忘却

- 新しい確定事項が古い記述を置き換える場合、追記だけでなく古い記述を更新する。
- 矛盾が解決していない場合は勝手に統合せず、`Unknown`として残す。
- 一時的な回避策には期限または見直し条件を付ける。
- 使われなくなった学習は削除候補として提示し、ユーザーの変更を無断で削除しない。

## 安全ルール

- APIキー、認証情報、個人情報、非公開データを保存しない。
- ユーザーの一時的な発言を永続的な好みとして断定しない。
- テスト未実施の修正を成功パターンとして昇格しない。
- 外部一般知識より、プロジェクト内の確認可能な事実を優先する。
- `.clinerules/`やSkillの意味を大きく変える場合は、変更理由と影響を明示する。

## 学習記録形式

```md
# Learning Review: <Topic>

## Source

## Candidates

| ID | Learning | Evidence | Confidence | Scope | Destination | Action |
|---|---|---|---|---|---|---|

## Promoted

## Kept As Candidates

## Rejected

## Replaced Or Stale

## Follow-up Validation
```

## 完了時の出力

- 抽出した学習候補数
- 昇格、保留、却下、置換した項目
- 更新ファイル
- 根拠とConfidence
- 次回の検証条件
